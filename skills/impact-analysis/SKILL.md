# 业务影响域分析

## 用途

开发前快速定位一个需求涉及哪些服务、表、MQ、调用链，以及改动时需要检查的联动点。
scenario.md 提供**业务架构地图**（不变的），具体代码位置通过**搜索关键词动态定位**（会变的）。

## 触发方式

- 用户输入 `/impact <关键词>` — 如 `/impact 标签`、`/impact 客户旅程`
- 用户说"查下业务影响域"、"这个需求涉及哪些地方要改"

## 执行步骤

### 1. 读取业务域索引

读取 `~/.claude/memory/areas/domains/index.md`，获取关键词→场景文件映射表。

### 2. 匹配业务域

从用户的需求描述或参数中提取关键词，与索引表匹配。
如果匹配到多个业务域，全部加载。
如果未匹配到任何业务域，提示用户该场景文档尚未创建。

### 3. 加载场景文档

读取匹配到的 `~/.claude/memory/areas/domains/{业务域}/scenario.md`。
获取：业务流程、服务职责、核心表、调用链、影响域 checklist。

### 4. 动态验证（关键步骤）

scenario.md 中的 checklist 每项都附带了**搜索关键词**。
根据用户需求，挑选相关的 checklist 项，用 Grep 在实际代码中验证：

```
对于每个相关的 checklist 项：
  1. 用搜索关键词在对应项目中 Grep
  2. 确认代码是否仍然存在、位置是否变化
  3. 找到实际的文件路径和类名
  4. 如果搜索不到，标记为「需确认：可能已重构」
```

**重要**：不要全部验证，只验证与本次需求直接相关的项（通常 3-8 项）。

### 5. 输出影响域分析

按以下结构输出：

```markdown
### 影响域分析：{业务域名}

**业务流程概要：**
[从 scenario.md 提取与本次需求相关的流程]

**本次改动直接涉及：**
- [服务] [实际文件路径:行号]（通过 Grep 验证）

**必须检查的联动点：**
- [ ] [联动项] → [实际文件路径]（已验证）
- [ ] [联动项] → 需确认：搜索 `关键词` 未命中，可能已重构

**核心表：**
- [仅列出与本次需求相关的表]

**MQ/跨服务调用：**
- [仅列出受影响的调用链]

**建议关注：**
[针对性的风险提示]
```

## 设计原则

- **scenario.md 记录不变的** — 业务流程、服务职责、表用途、调用链模式、checklist 类别
- **动态验证会变的** — 具体文件路径、类名、方法名，每次用搜索关键词从代码中定位
- **不全量验证** — 只验证与本次需求相关的 checklist 项，节省时间

## 注意事项

- 场景文档状态为 `todo` 的业务域尚未梳理，提示用户
- 只输出与用户需求相关的 checklist 项，不要全部罗列
- 如果用户的需求跨多个业务域，分别输出每个域的影响分析
- 验证失败的项不要删除，标记为「需确认」让用户判断
